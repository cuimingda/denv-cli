# ADR-001: 固定目录结构用于分层可维护性

## 决策标题
采用 `cmd/ + internal/{app,domain,infra}` 的分层结构，并通过 `internal/` 统一项目私有边界。

## 背景
项目在功能增长后出现了命令层和规则层混写的风险，未来接手者若直接改 `cmd` 到处调用运行时会放大耦合，难以安全演进。

## 可选方案
1. 不分层：保持所有逻辑在 `cmd`，以速度取代边界。  
2. 按 `cmd + internal` 两层实现：外部能力与业务规则混在 `internal` 内。  
3. 采用 `cmd + internal` 的三层职责：`domain` 纯规则、`infra` 外部适配、`service` 统一编排。

## 选择方案
选择方案 3。

## 选择理由
三层边界最小改造成本下明确 IO 边界，新增执行器或测试替身时可直接替换 `infra`；新功能改动集中在 `cmd` 与 `service`，降低对规则层污染。

## 未来可能调整条件
当出现跨语言服务拆分、需要 gRPC/HTTP 入站时，允许在 `internal/service` 上再新增适配入口，但保留 domain/infra 的职责隔离。
