# 项目概览（<1000 字）

## 系统目标
`denv` 是一个面向本地开发环境工具的 CLI，核心目标是：可读清晰地展示工具状态、识别过时版本、生成安装计划、执行安装更新操作，并通过统一错误语义给出可预测退出码。

## 系统边界
系统只处理本仓库内定义的工具清单（`internal/catalog.go`）与执行流程，不负责配置管理、容器编排、远端编译链托管或 CI/CD 平台改造。输出可见性与 CLI 用户体验是第一优先级，复杂依赖解析不在本系统范围。

## 核心模块
- `cmd/`：命令组装、参数校验、错误映射、用户输出。
- `internal/service.go`：应用层协调器，组装领域与基础设施能力。
- `internal/workflows.go`：核心用例入口（列表、过期、安装、更新）。
- `internal/domain/*`：纯业务规则与数据模型，不直接进行外部调用。
- `internal/infra/*`：外部命令执行、路径策略、运行时抽象、catalog 适配层。
- `docs/*`：结构与不变量知识沉淀，承载可传承性。

## 外部依赖
运行时外部依赖（如 shell 命令）全部通过 `internal/infra` 封装。命令发现、版本读取、安装动作通过接口注入，便于在测试和未来平台迁移时替换实现。业务层只依赖抽象能力，不直接依赖系统命令调用细节。

## 数据流
1. 用户命令从 `cmd/` 进入，完成参数解析与入口校验。  
2. `cmd` 调用 `internal/service` 构建用例输入。  
3. `internal/workflows` 调用 `internal/domain` 规则与状态模型。  
4. 与系统交互（文件、环境变量、进程执行）由 `internal/infra` 完成。  
5. 结果返回给 `cmd`，统一为文本/JSON 输出并转为退出码。  

关键不变量（见 `docs/invariants.md`）约束了命令顺序、JSON 契约、错误分支和退出语义的稳定性。

## 错误处理策略
- 参数错误：立即返回并输出使用帮助，退出码非 0。  
- 外部依赖缺失/不可用：尽早失败，停止后续步骤，返回可复用的失败语义。  
- 批量执行：遇到阻塞错误优先 fail-fast，避免部分成功造成不可预期状态。  
- 所有测试应覆盖失败路径，尤其是 `cli_contract` 与 `failure_contract`。
